AnyAPI ships a [Model Context Protocol](https://modelcontextprotocol.io) server so agents
can discover and run any API in the catalog without bespoke integrations. Point your MCP
client at one endpoint, sign in through OAuth when your client supports it, and your agent gets a
registry-driven tool surface that covers the entire catalog â€” no tool-per-API sprawl.

| | |
|---|---|
| **Endpoint** | `https://api.getanyapi.com/mcp` |
| **Transport** | Streamable HTTP |
| **Auth** | OAuth 2.1 for hosted clients; Bearer API key as a fallback |
| **Tools** | `list_apis`, `search_apis`, `get_api`, `quote_api`, `run_api`, `read_result`, `get_balance` |

## Prerequisites

- An AnyAPI account for OAuth sign-in, or an AnyAPI API key for clients without OAuth.
- A funded USD wallet. `run_api` bills your wallet per successful request; discovery and quoting do not charge the wallet.
- An MCP-capable client (Claude Code, Claude Desktop, Cursor, Codex, or any client that
  speaks Streamable HTTP).

<Note>
  Hosted Claude and ChatGPT connect with the URL only and handle OAuth sign-in and consent in
  the client. Do not paste a Bearer header into those hosted integrations. Use a Bearer API key
  only as a fallback for a client that cannot complete OAuth. The MCP endpoint requires
  authentication for every tool, including discovery. After authentication, discovery and quoting
  do not charge the wallet.
</Note>

## Tools

The server exposes seven generic tools. Discovery (`list_apis`, `search_apis`, `get_api`) and
`quote_api` are free; only `run_api` touches your wallet. `read_result` re-reads a prior run for
free, and `get_balance` reports what's left.

<AccordionGroup>
  <Accordion title="list_apis â€” browse the catalog" icon="list">
    Browse APIs as lightweight summaries, optionally filtered. Summaries carry the SKU,
    name, category, and USD pricing only - **no descriptions or schemas** - so listing the
    whole catalog stays token-cheap. To find an API by keyword (with descriptions), use
    `search_apis`; for one API's schemas, use `get_api`.

    **Input**

    | Field | Type | Required | Description |
    |---|---|---|---|
    | `query` | string | no | Free-text filter over name, slug, and description |
    | `category` | string | no | Category slug to filter by |

    **Returns** an array of API summaries (`id`, `slug`, `name`, `category`, `priceUsd`,
    `baseUsd`, `perItemUsd`).
  </Accordion>
  <Accordion title="search_apis â€” find an API by keyword" icon="magnifying-glass">
    Search APIs by keyword across name, slug, and description, returning matches **with
    their descriptions** (schemas omitted). This is the targeted lookup: use it to find the
    right SKU before reading its schema with `get_api`.

    **Input**

    | Field | Type | Required | Description |
    |---|---|---|---|
    | `query` | string | yes | Free-text search over name, slug, and description |
    | `category` | string | no | Category slug to narrow the search |
    | `limit` | integer | no | Cap on matches returned (default 25) |

    **Returns** `results` (API summaries including `description`) and `total`, the number
    of APIs that matched before `limit` was applied, so you know whether to narrow further.
  </Accordion>
  <Accordion title="get_api â€” inspect input and output schemas" icon="file-lines">
    Get the full definition of one API by SKU, including its normalized input and output
    JSON schemas.

    **Input**

    | Field | Type | Required | Description |
    |---|---|---|---|
    | `sku_id` | string | yes | The API SKU slug to describe |

    **Returns** the full API definition with input/output JSON schemas. Use the input
    schema to build a valid payload for `run_api`.
  </Accordion>
  <Accordion title="quote_api â€” estimate a call" icon="calculator">
    Estimate the USD cost of a call without executing it or charging your wallet.

    **Input**

    | Field | Type | Required | Description |
    |---|---|---|---|
    | `sku_id` | string | yes | The API SKU slug to price |
    | `input` | object | yes | The exact normalized payload you plan to pass to `run_api` |

    **Returns** a quote with `maxCostUsd`, `minCostUsd`, `baseUsd`, `perItemUsd`, `exact`, and
    `pricing`. It also returns `perItemUnit` when available. All monetary fields are in USD;
    `maxCostUsd` is the most the call can charge, while `minCostUsd` is the likely charge.
  </Accordion>
  <Accordion title="run_api â€” execute an API" icon="bolt">
    Execute an API by SKU with a normalized input payload. Requires a valid AnyAPI key.
    Billed to your USD wallet on success only.

    **Input**

    | Field | Type | Required | Description |
    |---|---|---|---|
    | `sku_id` | string | yes | The API SKU slug to execute |
    | `input` | object | yes | Normalized payload matching the API's input schema |
    | `fields` | string | no | Comma-separated keys (dotted paths like `author.name`) to keep on each result item. Trims the response, not the charge. |
    | `max_items` | integer | no | Cap the number of result rows returned; a `_truncated` note reports how many were withheld. Trims the response, not the charge. |
    | `summary` | boolean | no | Return only a structural outline (top-level keys and item counts) instead of the full data. Trims the response, not the charge. |

    The last three are **opt-in response-budget controls** â€” they keep a large result
    from flooding your context window. They change only what is returned to you, never
    what you are billed (`costUsd` reflects the full result the API produced).

    **Returns** the normalized `output`, the serving `provider` (always **AnyAPI**),
    `costUsd`, and `items` (the number of results charged for). Most APIs are priced
    per result â€” `baseUsd + perItemUsd Ă— items` â€” so cap results (and cost) with
    `limit` in your input; simple lookups are flat per request. Invalid input,
    insufficient funds, and total upstream failure are **not** charged.
  </Accordion>
  <Accordion title="read_result â€” re-read a saved result" icon="rotate">
    Re-read a recent `run_api` result for free. Provide its `result_id` and optional response-budget
    controls such as `fields`, `max_items`, `summary`, or `jq`.
  </Accordion>
  <Accordion title="get_balance â€” check your wallet" icon="wallet">
    Get the remaining wallet balance for the API key you authenticate with. Requires a
    valid AnyAPI key; never charges.

    **Input** â€” none.

    **Returns** `balanceUsd`, the remaining wallet balance in USD. Use it to confirm
    there are funds before a `run_api` call, or to surface the balance to a user.
  </Accordion>
</AccordionGroup>

## How an agent uses it

A typical flow chains the tools - discover, inspect, run:

<Steps>
  <Step title="Discover">
    The agent calls `search_apis` with a query (for example, "instagram") to find a matching
    SKU and read its description. (Use `list_apis` instead to browse the whole catalog as
    lightweight summaries.)
  </Step>
  <Step title="Inspect">
    It calls `get_api` with that `sku_id` to read the input schema and build a valid
    payload.
  </Step>
  <Step title="Run">
    It calls `run_api` with the `sku_id` and `input`. AnyAPI routes to a provider,
    normalizes the response, falls back automatically on error, and returns the result.
  </Step>
</Steps>

Because every API shares the same tools and one normalized schema convention, an
agent that learns this flow once can use the entire catalog.

### Worked example

Discover Reddit APIs, inspect one, and run it. Tool results carry both a JSON text block
and `structuredContent` with these exact shapes:

```json
// search_apis {"query": "reddit"} â†’ structuredContent
{
  "results": [
    {
      "id": "reddit.search",
      "slug": "reddit.search",
      "category": "social",
      "name": "Reddit Search",
      "description": "Search Reddit posts across all subreddits by query, normalized across providers with transparent failover.",
      "priceUsd": 0.003,
      "baseUsd": 0.003,
      "perItemUsd": 0
    }
  ],
  "total": 1
}
```

`list_apis` returns the same entries minus `description` (and wrapped in `scrapers`),
so browsing the full catalog stays token-cheap; reach for it when you want to enumerate
rather than search.

`perItemUsd` is `0` here, so this API is **flat per request** (`priceUsd`). APIs
priced per result report a non-zero `perItemUsd` and accept a `limit` in their
input â€” then `costUsd = baseUsd + perItemUsd Ă— items`.

```json
// get_api {"sku_id": "reddit.search"} â†’ structuredContent (schemas abridged)
{
  "scraper": {
    "slug": "reddit.search",
    "priceUsd": 0.003,
    "baseUsd": 0.003,
    "perItemUsd": 0,
    "inputSchema": {
      "type": "object",
      "required": ["query"],
      "properties": {
        "query":     { "type": "string" },
        "sort":      { "type": "string" },
        "timeframe": { "type": "string" },
        "after":     { "type": "string" }
      }
    },
    "outputSchema": { "type": "object" }
  }
}
```

## Errors

Tool failures come back as MCP tool errors with a short machine-readable prefix:

| Prefix | Meaning | Billed |
|---|---|---|
| `unauthorized` | Missing or invalid AnyAPI key | No |
| `sku_not_found` | The requested SKU does not exist | No |
| `invalid_input` | Payload did not match the API's input schema | No |
| `insufficient_credits` | Wallet balance too low â€” top up to run | No |
| `all_providers_failed` | Every provider for this API failed | No (released) |
| `no_providers` | The API has no enabled providers | No |
